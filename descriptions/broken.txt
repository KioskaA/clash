При сравнении описаний и анализа кода обнаружены следующие противоречия/неопределённости, которые требуют разрешения для корректной реализации бизнес-логики:

1. **Отношение к нулевому результату в играх клана (Clan Games)**  
   - В файле `desc2.txt` (пункт 3.4) сказано: «Если игрок не брал задания и по итогу имеет 0 очков — вопросов ноль».  
   - В файле `descmain.txt` в списке абсолютных показателей для исключения после периода указано: «Результат ИК менее 1000» — под это условие попадает и 0 очков, без каких-либо исключений.  
   - В текущем коде отсутствует логика обработки этого случая.  
   **Вопрос:** следует ли считать 0 очков допустимым, если игрок не участвовал (не брал задания), или же это всегда основание для исключения?

2. **Порог для досрочного исключения по очкам ИК**  
   - В `descmain.txt` (досрочное исключение) указан порог: «Результат ИК больше нуля, но меньше 200».  
   - В `desc2.txt` (пункт 3.3) сказано: «Если набрано 1–150 очков будут ОЧЕНЬ БОЛЬШИЕ ВОПРОСЫ. Спасти от кика … может только …».  
   - В коде порог не задан.  
   **Вопрос:** требуется унификация числового значения (200 или 150) для досрочного исключения.

3. **Критерий по атакам в рейдах**  
   - В `desc2.txt` (пункт 4.2) указано: «К соклану будут вопросы если он провел менее 3 атак».  
   - В `descmain.txt` и текущем коде отсутствует прямой порог; есть только «Пропуск всех рейдов» как основание для исключения и метрики «отношение атак к доступным».  
   - В интерфейсе (`PlayerFrame`) отображается общее число атак и доступных атак, но нет автоматического вывода о «вопросах» при менее чем 3 атаках.  
   **Вопрос:** следует ли ввести порог «менее 3 атак» как повод для предупреждения/санкций, или достаточно фиксировать только полный пропуск рейдов и использовать коэффициент участия?

Эти противоречия необходимо разрешить для корректной реализации логики исключения и расчёта рейтинга активности. Остальные аспекты, описанные в vision-файлах, либо уже отражены в структурах данных, либо запланированы для будущих версий.
